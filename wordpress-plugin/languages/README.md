# Translations

`orca-dam-picker.pot` is the extracted source template. Translators copy it to
`orca-dam-picker-{locale}.po` (e.g. `orca-dam-picker-nl_NL.po`) and fill in the
`msgstr` lines.

## Important: JS strings reference the BUILT bundles, not the source

`wp_set_script_translations()` loads a per-script JSON whose filename is
`orca-dam-picker-{locale}-{md5}.json`, where `{md5}` is the md5 of the **enqueued
script path relative to the plugin root** — e.g.
`md5("assets/build/editors/gutenberg.js")`.

`wp i18n make-json` derives that md5 from the `#:` reference comments in the
`.po`. Therefore the JS `#:` references in `orca-dam-picker.pot` /
`orca-dam-picker-nl_NL.po` deliberately point at the **build** files
(`assets/build/...`), not the `assets/src/...` sources. If they pointed at the
sources, the generated JSON filenames would not match what WordPress looks for
at runtime and the React UI would silently stay in English.

Strings used by the picker live in `assets/src/picker/Picker.js`, which webpack
bundles into both `assets/build/editors/gutenberg.js` and
`assets/build/editors/elementor.js`, so those strings carry **two** `#:`
references. Settings strings (`assets/src/settings/SettingsApp.js`) bundle into
`assets/build/settings.js`.

PHP strings keep normal `src/...` references (handled by the `.mo`, ignored by
`make-json`).

> Do **not** blindly overwrite the references with `wp i18n make-pot` from
> source — that re-introduces the source paths and breaks JS loading. If you
> regenerate the `.pot`, rewrite the JS `#:` references back to the matching
> `assets/build/*.js` bundle path(s) afterwards.

## Compiling .po → .mo (required for PHP runtime)

```bash
msgfmt languages/orca-dam-picker-nl_NL.po -o languages/orca-dam-picker-nl_NL.mo
```

## Splitting translations for the JS bundles

WordPress loads JS translations from per-script `.json` files generated by:

```bash
wp i18n make-json languages/ --no-purge
```

With the build-path references described above, this produces files named after
the md5 of each build bundle (e.g.
`orca-dam-picker-nl_NL-ece916f8b4be56d8e1da8b1a70692110.json` for
`assets/build/editors/gutenberg.js`), which `wp_set_script_translations()` loads
automatically as long as it is given the plugin's `/languages` directory as its
third argument.

The release CI workflow (`.github/workflows/wordpress-plugin-release.yml`) runs
`msgfmt` + `make-json` on every `wp-v*` tag before zipping. The compiled `.mo`
and `.json` files are git-ignored and only exist inside the released zip — a
source/dev checkout will show English until you compile them locally.
